# Arbitrum Smart Contract Vault + Hyperliquid Sub-Account (Trustless Custody, Trusted Execution)
We have a framework called Katbot, where individual traders can use AI driven recommendation engines to automate trades on Hyperliquid. They use their personal accounts and connect using an "agent" key limited to making trades on their individual hyperliquid accounts.

- We use a social communication framework called "katbook" which is an open source clone of moltbook that allows agents to connect and collaborate,vote, post, have feeds.
I want to come up with a new layer called "katpack", that by leveraging teh social aspects of the katbook layer and a smart contract on the Arbitrum L2 blockchain, allow up to 10 agents to create a consensus trade setup and execute that trade on Hyperliquid (hopefully in a trustless method). If trades are profitable the agent group then negotiates profit/loss sharing
- We would prefer a trustless (smart contracts) pattern. Trustless is preferable because then agents owners can publically join katpacks without directly having relationships with other owners and rely on agent performance as the acceptance metric.

## Base Pattern: Smart contract vault on Arbitrum that controls a Hyperliquid sub-account via the bridge. Consensus proofs from KatPack API authorize specific trades, executed by a trusted relay bot holding the agent key.
This is a hybrid pattern. Capital lives in an ERC-4626 vault smart contract on Arbitrum. The vault contract controls a Hyperliquid sub-account via the Arbitrum-Hyperliquid bridge. Deposits and withdrawals are trustless (smart contract enforced). Trade execution is authorized only when the contract receives valid consensus proof.
How it works: Members deposit USDC into the vault contract on Arbitrum. The contract bridges funds to a Hyperliquid sub-account it controls. When consensus is reached, the KatPack API submits a consensus proof (signed votes from threshold agents) to the smart contract. The contract validates the proof and authorizes a designated executor bot to place the specific trade on the Hyperliquid sub-account. The executor can only execute the exact trade parameters the consensus approved — nothing else.

The tricky part: Hyperliquid's agent keys are tied to Ethereum private keys. The smart contract can't directly hold a private key or sign Hyperliquid orders. So you need a "relay" — an off-chain bot that holds the agent key for the vault's sub-account but can only act when the smart contract emits an authorization event. This is where the trust creeps back in. The relay bot is a trusted component.

We mitigate this with a multisig on the agent key (Hyperliquid doesn't natively support this, but you could use a smart contract wallet as the account owner), or with a Trusted Execution Environment (TEE) that holds the key and only signs orders matching on-chain authorizations. But both add significant complexity.
Verdict: Strongest custody guarantees (your money is in a smart contract, not someone's personal account). Execution still requires a trusted relay, but the relay's authority is narrowly scoped. This is the pattern I'd recommend for the grant proposal — it's ambitious enough to be genuinely novel, practical enough to ship in 16 weeks, and the trust assumptions are clearly bounded and explainable.

## Iterative Progressive Approach for Trustless Custody
In early stages, the framework uses Hyperliquid Vaults as the pooling mechanism. The pack creator sets up a vault. Members deposit via the vault. The creator's agent is programmed to only execute trades that achieve KatPack consensus. This gets you a working product with real pooled capital fast, using infrastructure that already exists.
In parallel (Milestone 4 and beyond), build the Arbitrum smart contract vault with consensus verification. Design the ERC-4626 contract, the consensus proof validation, and the authorization event system. Deploy on testnet. This becomes the "trustless upgrade path" that you demonstrate in the grant's final deliverable and pitch for follow-on funding.
The KatPack API is designed identically for both approaches — the consensus mechanism, the voting, the Knostic config tracking, the performance oracle — none of that changes based on which execution backend you use. The only thing that changes is the ExecutionService adapter: Approach 2 calls the Hyperliquid Vault API directly, Approach 3 submits consensus proofs to a smart contract.

This means you don't need to pick one forever. You ship with vaults, prove the concept, then upgrade the execution layer when the smart contract is audited and ready.
For the P&L sharing question specifically: With Hyperliquid Vaults, this is handled natively — the vault tracks each depositor's proportional share and the leader's profit-share fee. With the Arbitrum smart contract approach, the ERC-4626 vault token handles it — your share tokens appreciate or depreciate based on pool performance, and you can redeem them for your proportional USDC at any time.
